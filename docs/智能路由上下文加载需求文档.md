# 智能路由上下文加载需求文档

版本：v0.6

## 一、业务需求

### 1. 背景

router 服务负责助手入口中的意图识别、任务规划、槽位收集、上下文管理和任务交接。随着业务 intent skill 增多，如果每次请求都加载全部业务知识，会导致上下文过大、意图干扰、成本升高和问题难定位。

本需求希望建立一套“按需加载、任务隔离、可释放、可观测”的上下文机制，保证系统在大量业务 skill 场景下仍能稳定处理单轮、多轮、单意图和多意图请求。

### 2. 目标

1. 支持 100 个以上业务 intent skill 的可发现和可路由。
2. 首轮仅基于 skill 摘要信息完成意图识别，不加载全部 skill 正文。
3. 命中意图后，只加载对应 skill 的业务规则。
4. 支持单轮单意图、单轮多意图、多轮补槽、任务切换和任务取消。
5. 支持多意图识别后形成多个任务；多任务按串行方式调度，同一时刻只处理一个当前任务，任务上下文互相隔离。
6. 支持长规则、复杂流程和低频资料按需加载。
7. 任务完成、取消、失败或 session 过期后，能及时释放相关上下文。
8. 提供 debug trace，便于观察识别、加载、规划、释放过程。
9. 保证流式和非流式返回的最终业务语义一致。

### 3. 业务边界

1. router 不硬编码具体业务意图、槽位规则和业务话术。
2. 不通过正则、关键词或模糊匹配兜底修补业务意图判断。
3. 不把所有 Markdown 业务知识一次性放入 prompt。
4. session 不保存 skill 正文、reference 正文或完整 prompt。
5. 多任务之间不得共享未授权的 skill 或 reference 正文。
6. 一个 `session_id` 对应且只对应一个用户，不允许跨用户复用状态。

## 二、功能需求

### 1. Skill 与 Intent 关系

1. 一个 skill 只能承载一个可派发业务 intent。
2. 一个 skill 不允许配置多个 `intent_code`。
3. 多意图输入必须拆分为多个独立 intent，并分别命中各自对应的 skill。
4. 如果多个业务 intent 需要共用规则，应拆成多个独立 skill，再通过公共 reference 复用资料。
5. skill 摘要用于意图识别；skill 正文用于槽位规则、业务边界、任务状态和 handoff 判断。

### 2. 单意图处理

当用户表达一个明确诉求时，系统应：

1. 基于 skill 摘要识别唯一 intent。
2. 校验 intent 与 skill 的对应关系。
3. 加载命中 skill 的业务规则。
4. 判断是否缺少必要槽位。
5. 槽位不足时发起补槽；槽位完整时进入可交接状态。
6. 任务结束后释放该任务相关上下文。

示例：“我要转账”应识别为转账 intent，并继续询问收款人、金额等缺失信息。

### 3. 多意图处理

当用户在一轮输入中表达多个诉求时，系统应：

1. 识别出多个独立 intent。
2. 为每个 intent 创建独立任务。
3. 每个任务只绑定自己的 skill、槽位和上下文引用。
4. 按用户表达顺序、业务优先级或 skill 规则确定串行处理顺序。
5. 同一时刻只能有一个 current task 进入补槽、澄清或交接处理。
6. 当前任务未结束前，后续任务保持等待状态，不并行推进业务处理。
7. 当前任务结束后，再继续调度下一个任务。
8. 单个任务结束时，只释放该任务上下文，不影响其他等待任务。

示例：“转账 200 给小明，再查余额”应拆为“转账”和“查余额”两个任务，而不是放进同一个 skill 内处理。

### 4. 多轮补槽与任务切换

当 session 中存在未完成任务时，系统应：

1. 将用户补充信息绑定到明确归属的任务。
2. 无法判断归属时发起澄清，不能随机写入某个任务。
3. 用户表达新诉求时，可以创建新任务或切换当前任务。
4. 用户取消某个任务时，只释放该任务上下文。
5. 用户取消全部任务时，释放当前 session 下全部任务上下文。
6. 已完成任务不再接收补槽，除非用户明确要求修改或重新办理。

### 5. Reference 按需加载

reference 用于承载长规则、复杂流程、大量槽位定义和低频案例。

1. reference 不参与全局意图识别。
2. 只有当前已命中 skill 明确声明的 reference 才允许加载。
3. reference 加载后只作用于当前任务。
4. 未授权、越权或路径不合法的 reference 请求必须被拒绝，并在 debug trace 中体现。

### 6. Session 生命周期

session 用于承载单个用户的身份绑定与空闲释放边界，不进入意图识别、补槽和任务调度核心逻辑。

1. 一个 `session_id` 只能绑定一个用户。
2. session 默认空闲 30 分钟过期。
3. 空闲时间从最近一次有效请求或状态更新开始计算。
4. session 过期表示该会话下没有可继续的任务运行态；必须清除该 key 下的全部任务内存，包括槽位、任务列表、当前任务、上下文引用和轻量任务状态。
5. session 过期后再次收到相同 `session_id` 的请求，应按新 session 处理，不继承历史任务或上下文。
6. session 过期清理只影响当前 session，不影响其他用户或其他 session。
7. session 不得保存或表达业务任务状态；任务状态只属于独立的任务运行态 `task_list` 和 `current_task`。
8. router 核心 planner 输入应是任务运行态，不应直接依赖 session 生命周期对象。

### 7. 可观测性

debug trace 开启时，应能观察以下关键阶段：

1. 请求进入与 session 读取。
2. session 用户绑定与过期检查。
3. skill 摘要索引生成。
4. 单意图或多意图识别结果。
5. 任务创建、更新、切换和结束。
6. skill 正文加载。
7. reference 允许、加载或拒绝。
8. 槽位判断、澄清和交接结果。
9. 上下文释放和 session 清理。

非 debug 模式下，只输出正常业务消息和完成信号。

### 8. 任务推荐

前端可在请求中传入推荐任务列表，由用户决定是否采纳。

1. 推荐任务列表由前端按业务场景生成，每条包含至少一个 intent 描述。
2. 用户可选择全部执行、部分执行或不执行推荐任务。
3. 用户选择执行的推荐任务，按正常多意图流程拆分为独立任务并串行调度。
4. 用户不选择任何推荐任务而表达其他诉求时，按正常意图识别流程处理，推荐列表不干扰识别结果。
5. 用户未对推荐列表做出选择时，推荐列表不影响后续轮次的意图识别。

示例：前端传入推荐列表 ["查余额", "转账"]，用户回复"全部办理"则创建两个任务串行执行；用户回复"只查余额"则只创建查余额任务；用户回复"我要还款"则忽略推荐列表，按还款意图处理。

## 三、接口需求

本阶段需求范围内保留 4 个 HTTP 接口：业务主链路接口 2 个，探活接口 2 个。除探活外，不要求提供额外辅助调试接口。

### 1. 消息入口接口

`POST /api/v1/message`

用途：接收用户自然语言输入，完成意图识别、任务创建或更新、补槽、任务切换、上下文加载和业务协议输出。

请求要求：

1. 必须包含 `sessionId` 和 `txt`。
2. `sessionId` 必须绑定单个用户；当前默认使用 `cust_no` 作为用户绑定标识。
3. 支持 `stream` 控制流式或非流式输出。
4. 支持 `debugTrace` 控制是否输出调试链路。
5. 支持 `executionMode` 区分仅路由规划或继续交接执行。
6. 可接收推荐任务列表、当前展示态和上游配置变量，但这些信息只能影响当前路由判断，不得污染其他 session 或任务。
7. 推荐任务列表为可选字段，不传时不影响正常意图识别流程。列表中每条应包含可识别的 intent 描述。

响应要求：

1. 流式模式通过 SSE 输出 `message` 和 `done`。
2. debug 模式可额外输出 trace，但不改变最终业务语义。
3. 非流式模式返回最终业务帧。
4. 可识别意图时，应先产生意图识别结果，再进入补槽、澄清或交接状态。
5. 多意图场景应返回可表达多个任务的结构，且任务顺序与规划顺序一致。

### 2. 任务完成回调接口

`POST /api/v1/task/completion`

用途：接收下游助手或执行系统对某个任务的完成确认，推动 router 更新任务状态、释放上下文，并在多任务场景下继续调度后续任务。

请求要求：

1. 必须包含 `sessionId`、`taskId` 和 `completionSignal`。
2. `completionSignal=1` 表示阶段性完成确认。
3. `completionSignal=2` 表示最终完成确认。
4. 只能更新当前 session 下存在且未结束的任务。
5. 已完成、已取消或不存在的任务不得被重复确认为有效完成。

响应要求：

1. 支持流式和非流式输出。
2. 单任务最终完成后，应返回 completed 状态并释放该任务上下文。
3. 多任务场景下，当前任务完成后，可在同一次响应中继续推动下一个 ready 任务。
4. 完成确认不得影响其他用户或其他 session 的任务状态。

### 3. 探活接口

探活接口用于部署、负载均衡和运行状态检查，不承载业务处理。

1. `GET /healthz`：基础存活检查，只判断服务进程是否可响应。
2. `GET /readyz`：服务就绪检查，用于判断服务是否具备接收业务请求的条件。

## 四、技术需求

### 1. 分层上下文

系统应至少区分三类上下文：

1. Agent Context：全局行为约束、协议边界和输出纪律，每次请求默认可用。
2. Skill Metadata：轻量 skill 摘要，用于意图识别，可支持大规模 skill 列表。
3. Skill Body 与 Reference：命中任务后按需加载，只在授权任务范围内生效。

### 2. 任务与上下文隔离

1. 每个识别出的 intent 应对应一个独立任务。
2. 每个任务应维护独立的槽位状态、任务状态和上下文引用。
3. 上下文引用只保存必要标识，不保存 Markdown 正文。
4. 后续轮次可根据上下文引用重新加载所需 skill 或 reference。
5. 任务完成、取消或失败后，必须释放该任务上下文引用。

### 3. 安全约束

1. 文件加载必须受配置根目录和 skill 目录限制。
2. 禁止路径穿越。
3. 禁止根据用户输入直接拼接文件路径。
4. 生产环境默认不暴露 prompt、skill body 或 reference body 明文 trace。
5. 同一 `session_id` 被不同用户使用时，必须拒绝或重新分配 session，不能合并状态。

## 五、验收标准

1. 100 个以上 skill 场景下，首轮识别只依赖 skill 摘要。
2. 单意图请求能识别 intent、加载对应 skill，并完成补槽或交接判断。
3. 多意图请求能拆分为多个任务，且每个任务对应独立 skill。
4. 一个 skill 配置多个 intent 时应被视为不合法。
5. 多轮补槽能正确归属任务；归属不明时必须澄清。
6. 多意图形成多个任务后，应按串行顺序处理，同一时刻只有一个 current task。
7. 未授权 reference 不能加载。
8. 任务结束后，对应上下文引用被释放。
9. session 空闲 30 分钟后过期，并清理该 session 全部内存状态。
10. 过期后的相同 `session_id` 请求按新 session 处理。
11. session 过期清理不影响其他 session。
12. 流式和非流式返回的最终业务语义一致。
13. debug trace 能看清识别、加载、规划、释放和过期清理关键过程。
14. `/api/v1/message` 能覆盖单意图、多意图、补槽、澄清、切换和取消场景。
15. `/api/v1/task/completion` 能覆盖单任务完成、多任务串行继续调度和重复完成拒绝场景。
16. 探活接口保持可访问；除探活外，不验收其他辅助调试接口。
17. 前端传入推荐任务列表时，用户选择全部执行、部分执行或不执行均能正确响应。
18. 用户不采纳推荐而表达其他意图时，推荐列表不干扰正常意图识别。

## 六、框架选型说明

### 1. DeepAgents 调研结论

DeepAgents 的渐进式上下文加载思路值得借鉴，但当前服务不建议整体采用 DeepAgents runtime，也不启用 subagent。

主要判断：

1. DeepAgents 更适合个人 CLI、开发者 agent harness 或长任务自动化场景。
2. 当前项目是多人在线 server，需要更稳定的性能、session 隔离、协议稳定性和确定性流程。
3. 面客服务应由服务端状态机控制 intent 识别、task 创建、current task 推进、上下文加载和释放，而不是让 agent 自主决定执行路径。
4. subagent 会增加额外模型调用、上下文复制和调度开销，不符合当前性能目标。

### 2. 可借鉴能力

1. skill metadata 与 skill body 分层。
2. 首轮只加载轻量 skill 摘要，命中后再加载 skill 正文。
3. 长规则、复杂流程和低频资料拆成 reference。
4. reference 由当前 skill 显式声明，并在需要时按需加载。
5. 上下文加载过程应可观测，便于 debug trace 展示。

### 3. 不采用能力

1. 不启动 subagent。
2. 不做多 agent 并行任务拆解。
3. 不使用 agent 自主长期持有上下文。
4. 不把业务流程控制权交给 agent runtime。
5. 不依赖个人 CLI 式本地文件状态作为 server session 状态。

### 4. 当前技术路线

当前服务采用单 runtime、单 planner、服务端状态机驱动。

请求处理原则：

1. 每次请求只处理一个 current task。
2. 多意图识别后可以形成多个 task，但按串行顺序推进。
3. LLM 输出结构化规划结果，服务端负责校验 skill、intent、task 和状态迁移。
4. skill/reference 加载由 context lease 控制。
5. task 完成、取消、失败或 session 过期后，释放对应 context lease。
6. session 只保存轻量状态和上下文引用，不保存 skill body、reference body 或完整 prompt。

目标模型：

```text
DeepAgents progressive loading 思路
  + 自研 deterministic router state machine
  + 无 subagent
  + context lease 生命周期控制
  + 多任务串行调度
```
