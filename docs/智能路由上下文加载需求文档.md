# 智能路由上下文加载需求文档

版本：v0.2

## 一、业务需求

### 1. 背景

router 服务负责助手入口中的意图识别、任务规划、槽位收集、上下文管理和任务交接。随着业务 intent skill 增多，如果每次请求都加载全部业务知识，会导致上下文过大、意图干扰、成本升高和问题难定位。

本需求希望建立一套“按需加载、任务隔离、可释放、可观测”的上下文机制，保证系统在大量业务 skill 场景下仍能稳定处理单轮、多轮、单意图和多意图请求。

### 2. 目标

1. 支持 100 个以上业务 intent skill 的可发现和可路由。
2. 首轮仅基于 skill 摘要信息完成意图识别，不加载全部 skill 正文。
3. 命中意图后，只加载对应 skill 的业务规则。
4. 支持单轮单意图、单轮多意图、多轮补槽、任务切换和任务取消。
5. 支持多个任务在同一 session 中并存，且上下文互相隔离。
6. 支持长规则、复杂流程和低频资料按需加载。
7. 任务完成、取消、失败或 session 过期后，能及时释放相关上下文。
8. 提供 debug trace，便于观察识别、加载、规划、释放过程。
9. 保证流式和非流式返回的最终业务语义一致。

### 3. 业务边界

1. router 不硬编码具体业务意图、槽位规则和业务话术。
2. 不通过正则、关键词或模糊匹配兜底修补业务意图判断。
3. 不把所有 Markdown 业务知识一次性放入 prompt。
4. session 不保存 skill 正文、reference 正文或完整 prompt。
5. 多任务之间不得共享未授权的 skill 或 reference 正文。
6. 一个 `session_id` 对应且只对应一个用户，不允许跨用户复用状态。

## 二、功能需求

### 1. Skill 与 Intent 关系

1. 一个 skill 只能承载一个可派发业务 intent。
2. 一个 skill 不允许配置多个 `intent_code`。
3. 多意图输入必须拆分为多个独立 intent，并分别命中各自对应的 skill。
4. 如果多个业务 intent 需要共用规则，应拆成多个独立 skill，再通过公共 reference 复用资料。
5. skill 摘要用于意图识别；skill 正文用于槽位规则、业务边界、任务状态和 handoff 判断。

### 2. 单意图处理

当用户表达一个明确诉求时，系统应：

1. 基于 skill 摘要识别唯一 intent。
2. 校验 intent 与 skill 的对应关系。
3. 加载命中 skill 的业务规则。
4. 判断是否缺少必要槽位。
5. 槽位不足时发起补槽；槽位完整时进入可交接状态。
6. 任务结束后释放该任务相关上下文。

示例：“我要转账”应识别为转账 intent，并继续询问收款人、金额等缺失信息。

### 3. 多意图处理

当用户在一轮输入中表达多个诉求时，系统应：

1. 识别出多个独立 intent。
2. 为每个 intent 创建独立任务。
3. 每个任务只绑定自己的 skill、槽位和上下文引用。
4. 按用户表达顺序、业务优先级或 skill 规则确定处理顺序。
5. 可完成的任务进入可交接状态；缺槽任务进入补槽状态。
6. 单个任务结束时，只释放该任务上下文，不影响其他未完成任务。

示例：“转账 200 给小明，再查余额”应拆为“转账”和“查余额”两个任务，而不是放进同一个 skill 内处理。

### 4. 多轮补槽与任务切换

当 session 中存在未完成任务时，系统应：

1. 将用户补充信息绑定到明确归属的任务。
2. 无法判断归属时发起澄清，不能随机写入某个任务。
3. 用户表达新诉求时，可以创建新任务或切换当前任务。
4. 用户取消某个任务时，只释放该任务上下文。
5. 用户取消全部任务时，释放当前 session 下全部任务上下文。
6. 已完成任务不再接收补槽，除非用户明确要求修改或重新办理。

### 5. Reference 按需加载

reference 用于承载长规则、复杂流程、大量槽位定义和低频案例。

1. reference 不参与全局意图识别。
2. 只有当前已命中 skill 明确声明的 reference 才允许加载。
3. reference 加载后只作用于当前任务。
4. 未授权、越权或路径不合法的 reference 请求必须被拒绝，并在 debug trace 中体现。

### 6. Session 生命周期

session 用于承载单个用户的多轮业务状态。

1. 一个 `session_id` 只能绑定一个用户。
2. session 默认空闲 30 分钟过期。
3. 空闲时间从最近一次有效请求或状态更新开始计算。
4. session 过期后，必须清除该 session 下的全部内存状态，包括槽位、任务列表、当前任务、上下文引用和轻量任务状态。
5. session 过期后再次收到相同 `session_id` 的请求，应按新 session 处理，不继承历史任务或上下文。
6. session 过期清理只影响当前 session，不影响其他用户或其他 session。

### 7. 可观测性

debug trace 开启时，应能观察以下关键阶段：

1. 请求进入与 session 读取。
2. session 用户绑定与过期检查。
3. skill 摘要索引生成。
4. 单意图或多意图识别结果。
5. 任务创建、更新、切换和结束。
6. skill 正文加载。
7. reference 允许、加载或拒绝。
8. 槽位判断、澄清和交接结果。
9. 上下文释放和 session 清理。

非 debug 模式下，只输出正常业务消息和完成信号。

## 三、技术需求

### 1. 分层上下文

系统应至少区分三类上下文：

1. Agent Context：全局行为约束、协议边界和输出纪律，每次请求默认可用。
2. Skill Metadata：轻量 skill 摘要，用于意图识别，可支持大规模 skill 列表。
3. Skill Body 与 Reference：命中任务后按需加载，只在授权任务范围内生效。

### 2. 任务与上下文隔离

1. 每个识别出的 intent 应对应一个独立任务。
2. 每个任务应维护独立的槽位状态、任务状态和上下文引用。
3. 上下文引用只保存必要标识，不保存 Markdown 正文。
4. 后续轮次可根据上下文引用重新加载所需 skill 或 reference。
5. 任务完成、取消或失败后，必须释放该任务上下文引用。

### 3. 安全约束

1. 文件加载必须受配置根目录和 skill 目录限制。
2. 禁止路径穿越。
3. 禁止根据用户输入直接拼接文件路径。
4. 生产环境默认不暴露 prompt、skill body 或 reference body 明文 trace。
5. 同一 `session_id` 被不同用户使用时，必须拒绝或重新分配 session，不能合并状态。

## 四、验收标准

1. 100 个以上 skill 场景下，首轮识别只依赖 skill 摘要。
2. 单意图请求能识别 intent、加载对应 skill，并完成补槽或交接判断。
3. 多意图请求能拆分为多个任务，且每个任务对应独立 skill。
4. 一个 skill 配置多个 intent 时应被视为不合法。
5. 多轮补槽能正确归属任务；归属不明时必须澄清。
6. 多任务并存时，单个任务结束不影响其他任务上下文。
7. 未授权 reference 不能加载。
8. 任务结束后，对应上下文引用被释放。
9. session 空闲 30 分钟后过期，并清理该 session 全部内存状态。
10. 过期后的相同 `session_id` 请求按新 session 处理。
11. session 过期清理不影响其他 session。
12. 流式和非流式返回的最终业务语义一致。
13. debug trace 能看清识别、加载、规划、释放和过期清理关键过程。

## 五、框架选型说明

当前阶段不重点展开框架调研。优先完成需求闭环和行为验收，框架选择只需保证后续可兼容渐进式上下文加载、任务状态管理和按需 reference 加载。
